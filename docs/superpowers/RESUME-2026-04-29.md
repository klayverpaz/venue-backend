### 2) Smoke script que percorre todos os endpoints e popula o banco

Objetivo: exercitar o fluxo end-to-end real (não mais SQLite/in-memory) — registrar admin, criar `ResourceType`, registrar owner + customer, criar Resource, requestar booking, aprovar, ratear, listar tudo. Vai surfaçar bugs reais que os testes em SQLite não pegaram (ex.: tipos Postgres-only, comportamento do advisory lock, `btree_gist` constraint, JSON columns).

Sugestão:

- `scripts/smoke_full_flow.py` (ou o nome que preferir) usando `httpx.AsyncClient` apontando para `http://localhost:8000`.
- Sequência:
  1. Login como admin (admin já existente do bootstrap).
  2. `POST /v1/admin/resource-types` → cria "Football Field" com schema de atributos básicos.
  3. `POST /v1/auth/register` (role=owner) + login.
  4. `POST /v1/auth/register` (role=customer) + login.
  5. `POST /v1/me/resources` (owner) → cria um campo. Atenção a `operating_hours`: keys lowercase (`monday`, etc.), times no formato `"HH:MM"`, `slot_duration_minutes ∈ {30,45,60,90,120}`.
  6. `POST /v1/me/resources/{id}/publish` (owner).
  7. `GET /v1/resources` (público, sem auth) → confirma que aparece com `rating_avg=null, rating_count=0`.
  8. `POST /v1/me/bookings` (customer) → request booking. Slot 1-2 dias no futuro, dentro do operating_hours, alinhado ao slot grid.
  9. `POST /v1/me/bookings/{id}/approve` (owner).
  10. (Opcional, requer manipulação de tempo) Avançar slot para o passado via SQL direto, depois `POST /v1/me/bookings/{id}/rating` (customer).
  11. `GET /v1/owners/{owner_slug}/resources/{resource_slug}/ratings` (público) → confirma rating no listing.
  12. `GET /v1/me/notifications` (owner) → confirma BOOKING_REQUESTED + BOOKING_APPROVED entregues.
  13. `GET /v1/admin/users` (admin) → listing.

- Cada step com `assert response.status_code == ...` e log do payload pra diagnosticar erros.
- Quando algum step falhar, debugar/corrigir o bug e re-rodar.

### Áreas de bug provável (suspects do final-reviewer e da minha intuição)

- **Postgres `btree_gist` extension + EXCLUDE constraint** — só foi exercitada em SQLite (que não tem essas features). Primeira aprovação de booking em Postgres pode revelar problemas de configuração da migration.
- **Pydantic + `Decimal` + Postgres `numeric(3,1)` round-trip** — Plan 09 corrigiu serialização JSON, mas o fluxo banco→agg→JSON em Postgres pode ter edge cases não testados (precisão do `AVG`, `quantize` com `ROUND_HALF_EVEN`).
- **Timezone handling** — testes usam SQLite que faz strip de tzinfo; Postgres preserva. Se algum lugar do código depende do strip pra "normalizar", quebra. O helper `_ensure_utc` em todos os repos é a defesa contra isso, mas aplica apenas no read path.
- **JSON columns** — `Booking.status_history`, `Resource.operating_hours`, `Resource.pricing_rules`, `ResourceType.attribute_schema`. SQLite e Postgres serializam JSON levemente diferente; tipos Decimal dentro de JSON podem dar problema.
- **Redis pool no boot** — `app/main.py` lifespan exige Redis ativo. Sem Redis no compose, app não sobe. Verifique que está no compose.
- **Argon2 + cost params** — `.env.example` tem cost moderado. Em container resource-limited pode demorar. Não é bug, só lentidão.
